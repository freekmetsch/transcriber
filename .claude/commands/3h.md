---
description: Planning workflow - intent gathering, architecture discovery, and phased execution plans
---

> See `.claude/commands/__common.md` for shared context.

Use `/3h` for:
- Existing feature lists that need refinement into an execution-ready plan.
- A single large vague idea.
- A mixed dump of ideas that needs intent capture, scope control, and architecture-aware planning.
- Work that is intended to execute in this chat window or a clearly defined follow-up window.

Use `/list` instead when the user is only dumping backlog ideas and is not ready for execution-ready phasing yet.

## 0. Communication Mode

Default `No Decision Needed`. Switch to `Decision Required` for the final planning synthesis and decision gates.

## 1. Intake and Input Mode Detection

Identify the planning input mode:
1. Existing active artifact (`docs/feature-lists/*.md`)
2. Raw idea dump
3. Hybrid (artifact + new ideas)

Preserve the user's raw intent before restructuring it.

## 2. Intent Gathering and Ambiguity Resolution

Create an intent brief:
- Objective (what outcome matters most)
- Constraints (time, risk, architecture, compatibility, cost)
- Success criteria (what must be true to call this done)
- Scope boundaries (in-scope vs out-of-scope)

When instructions are ambiguous or judgment-heavy:
- Provide 2-3 concrete options.
- State tradeoffs and risk for each option.
- Recommend one option.
- Ask only minimum clarifying questions required to avoid execution risk.

## 3. Architecture Discovery (What Exists vs What Is Missing)

Perform discovery before proposing execution steps:
- Read `docs/FLOW.md` for the end-to-end message pipeline, action dispatch, error handling, and file structure. If `Last verified` date is >7 days old, spot-check key function names (e.g., `route`, `handle_text`, `maybe_consolidate`) still exist before trusting pipeline descriptions.
- Read `spec.md` and relevant active artifacts.
- Map likely touched files: `bot.py`, `config.py`, `processor.py`, `vault.py`, `memory.py`, `transcriber.py`, `models.py`, `prompts/`.
- Identify reusable patterns that already exist.
- Identify true gaps that require new implementation.

Research requirements:
- Use `#context7` for all framework/library behaviors and APIs referenced by the plan (python-telegram-bot, Pydantic AI, Pydantic, etc.).

## 4. Gold Standard and Technical Debt Guardrail

For every option, evaluate explicitly:
- Gold Standard alignment: modern, idiomatic Python patterns for the current stack.
- Technical debt risk: what short-term shortcuts could create future rework.
- Sustainability: maintainability, extensibility, and operational reliability over time.

Reject options that are faster short term but create avoidable structural debt unless the user explicitly approves that tradeoff.

## 5. Optioning and Recommendation

Present 2-3 viable implementation approaches.
For each approach include:
- Scope surface
- Architectural fit
- Risk/regression profile
- Sustainability/maintenance cost
- Why it may or may not fit current constraints

Recommend one approach with explicit rationale.

## 6. Decision Synthesis for User

Before hard stop, provide a user-friendly decision brief:
- 5-8 line plain-language summary of the recommended path.
- Top tradeoffs and why this recommendation wins.
- What is intentionally out of scope right now.
- Clear decision questions that require user confirmation.
- Use `Decision Required` mode format with 2-3 options and explicit reply choices.

Decision questions must be concrete and answerable (for example: option A/B/C, phase split choice, or risk tolerance choice).

## 7. Size, Risk Tiers, and Context-Window Strategy

Classify task size per `__common.md` Context Budget.

Classify risk tier and apply controls per `__common.md` Risk Tiers.

Define phases with execution boundaries:
- Phase goal
- In-scope files/systems
- Validation expectations
- Exit criteria
- Dependencies/blockers

For each phase, decide context strategy:
- `single-window`: can be executed safely in one agent context window
- `multi-window`: requires handoff due to scope, risk, or context volume

Define agent-switch checkpoints when `multi-window`:
- Switch at phase boundaries, not mid-phase.
- Produce a handoff packet:
  - Completed work summary
  - Files touched
  - Open decisions
  - Pending validations
  - Next-step command for the next agent

## 8. Feature Ticket Schema (Required)

Each execution ticket in the active feature list must include minimum required fields:
- Scope (in/out)
- Target files/paths
- Risk tier (`R0`-`R3`)
- Verification steps (using commands from `__common.md` Verification Commands)
- Rollback plan

Add these fields when context warrants (medium/large scope, uncertainty, or handoff risk):
- Impact score (1-5)
- Effort estimate (S/M/L or hours)
- Confidence (High/Med/Low)
- Dependencies/blockers

## 9. Final Planning Output (Execution-Ready)

Create or update a feature list artifact that an execution agent can run:
- `docs/feature-lists/FEATURE_LIST_[NAME].md`

Required core sections (always):
1. Problem framing
2. Scope (in/out)
3. Option comparison + chosen approach
4. Phase plan with context-window strategy
5. Risk tier and verification matrix
6. Open questions and decision gates

Include conditional sections when relevant:
1. Existing system inventory (when touching existing subsystems)
2. Agent handoff checkpoints (when `multi-window`)
3. Ticket schema completeness check (when more than one execution ticket)
4. Rollback and rollout strategy (mandatory for `R2`/`R3`; optional for `R0`/`R1`)

## 10. Hard Stop

Do not execute implementation in this workflow.
Wait for explicit user instruction to execute (for example `/go`).
